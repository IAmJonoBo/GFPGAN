    Contributing to GFPGAN¶

    We welcome contributions to GFPGAN! This guide will help you get started with developing and contributing to the project.

    Getting started¶

    Development environment setup¶

    1. Fork and clone the repository
    git clone https://github.com/YOUR_USERNAME/GFPGAN.git
cd GFPGAN
    1. Create a virtual environment
    python -m venv venv
source venv/bin/activate  # On Windows: venv\Scripts\activate
    1. Install in development mode
    pip install -e ".[dev,metrics,web]"

    This installs: - dev: Linting, formatting, and testing tools - metrics: Quality evaluation dependencies - web: Web interface dependencies

    1. Set up pre-commit hooks
    pre-commit install

    Development workflow¶

    1. Create a feature branch
    git checkout -b feature/your-feature-name

    1. Make your changes

    2. Write clear, well-documented code

    3. Follow existing code style and patterns

    4. Add tests for new functionality

    5. Test your changes

    # Run linting and formatting
ruff check .
black .

# Run tests
pytest tests_light/  # Quick tests
pytest tests/        # Full test suite
    1. Commit and push
    git add .
git commit -m "feat: add your feature description"
git push origin feature/your-feature-name

    1. Create a pull request

    2. Use a clear, descriptive title

    3. Explain what your changes do and why
    4. Reference any related issues
    5. Ensure all checks pass

    Code style and standards¶

    Python code style¶

    We use automated formatting and linting:

    • Black: Code formatting
    • Ruff: Fast linting and import sorting
    • Type hints: Required for public APIs
    # Format code
black .

# Check linting
ruff check .

# Fix auto-fixable issues
ruff check --fix .

    Commit message format¶

    We follow Conventional Commits:

    type(scope): description

[optional body]

[optional footer(s)]

    Types: - feat: New feature - fix: Bug fix - docs: Documentation changes - refactor: Code refactoring - test: Adding or updating tests - chore: Maintenance tasks

    Examples: 

    feat(cli): add dry-run mode for batch processing
fix(web): handle missing face detection gracefully
docs(guides): update hardware requirements

    Documentation¶

    All user-facing features should include documentation:

    • API functions: Docstrings with examples
    • CLI commands: Help text and guide updates
    • New features: Usage guides and examples

    Testing¶

    Test structure¶

    tests/                 # Full test suite
├── test_gfpgan_arch.py
├── test_models.py
└── ...

tests_light/           # Quick tests for CI
├── test_basic.py
└── ...

    Running tests¶

    # Quick tests (recommended for development)
pytest tests_light/ -v

# Full test suite (for comprehensive validation)
pytest tests/ -v

# Test specific functionality
pytest tests/test_gfpgan_model.py -v

# Test with coverage
pytest tests_light/ --cov=gfpgan --cov-report=html

    Writing tests¶

    • Test both success and failure cases
    • Use clear, descriptive test names
    • Include edge cases and boundary conditions
    • Mock external dependencies when needed

    Example: 

    def test_gfpgan_infer_with_valid_image():
    """Test that GFPGAN inference works with valid input image."""
    # Test implementation
    pass

def test_gfpgan_infer_with_invalid_format():
    """Test that GFPGAN handles invalid image formats gracefully."""
    # Test implementation
    pass

    Pull request checklist¶

    Before submitting a pull request, ensure:

    • [ ] Code quality
    • [ ] Code follows style guidelines (Black + Ruff)
    • [ ] All tests pass (pytest tests_light/)

    • [ ] No linting errors (ruff check .)

    • [ ] Documentation

    • [ ] Docstrings added for new functions/classes
    • [ ] User guides updated if needed

    • [ ] CHANGELOG.md updated for user-facing changes

    • [ ] Testing

    • [ ] New tests added for new functionality
    • [ ] Existing tests still pass

    • [ ] Edge cases considered

    • [ ] Compatibility

    • [ ] Changes don't break existing API
    • [ ] Backward compatibility maintained
    • [ ] Cross-platform compatibility verified

    Development tasks¶

    Running the documentation site locally¶

    # Install docs dependencies
pip install -e ".[docs]"

# Serve docs locally
mkdocs serve

# Build static site
mkdocs build

    Building and testing packages¶

    # Build wheel
python -m build

# Test installation
pip install dist/gfpgan-*.whl

    Benchmarking changes¶

    # Run benchmark suite
python bench/run_bench.py --input inputs/samples/ --output bench_results/

# Compare with baseline
python bench/compare_results.py --baseline bench_baseline.json --current bench_results.json

    Release process¶

    1. Update version numbers
    2. VERSION file
    3. pyproject.toml

    4. gfpgan/version.py

    5. Update CHANGELOG.md

    6. Move items from "Unreleased" to new version section

    7. Follow Keep a Changelog format

    8. Create release PR

    9. Title: "Release v1.2.3"
    10. Include changelog highlights

    11. Ensure all tests pass

    12. Tag and release 

      git tag v1.2.3
git push origin v1.2.3

    13. Deploy documentation 

      mike deploy --push --update-aliases 1.2.3 latest

    Getting help¶
